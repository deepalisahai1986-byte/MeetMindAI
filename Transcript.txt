It's slow, error-prone, and honestly a bottleneck for the business. What I'd like us to explore today is how Appian's native AI Skills — specifically the Document Extraction skill — can automate that intake layer. The goal is to extract structured data from unstructured PDFs and hand it off cleanly into our Appian records.

**Whales:** That aligns well with the Q3 delivery milestone we committed to the stakeholders. Deepali, just to clarify for the team — are we talking about a fully automated pipeline or is there still a human-in-the-loop step?

Think of it as a confidence-scored extraction where anything below 85% confidence gets flagged for manual review. We want to build trust in the system before we go fully hands-off.

**Rahul:** That actually makes my job a lot clearer. So on the UI side, I'm essentially designing a review interface — something like a split-screen where the original document is on one side and the extracted fields are on the other? The reviewer can then accept, edit, or reject each field?

One thing I want to flag though — Appian's Document Extraction AI Skill returns a JSON payload with field names, values, and confidence scores. We need to decide how we map those field names to human-readable labels in the UI. Because raw JSON keys like `vendor_tax_id` or `invoice_net_amount` aren't what a reviewer wants to see.

**George:** That's a good point. Should we create a field mapping configuration? Or is that something we hardcode initially and iterate on?

**Rahul:** I'd recommend a configuration-driven approach from the start. If we hardcode, we'll be back here in Sprint 4 fixing it when the ops team asks for a different field label or a new document type gets added.

**Whales:** I agree, but let's be realistic about timelines. Rahul, if we go config-driven from day one, what's the effort delta compared to hardcoding for the MVP?

**Rahul:** Honestly, maybe an extra day and a half of work — building a simple admin screen where you map AI skill output fields to display labels. It's not massive, but it has to be scoped in.

**Whales:** Deepali, your call on that one.

**Deepali:** Let's do it right. Add it to the backlog as a must-have for Sprint 1, not a nice-to-have. I don't want tech debt to slow us down in Sprint 3.

**Deepali:** I've reviewed the documentation. Appian's Document Extraction uses a trained model underneath — you define a Document Type, specify the fields you want extracted, and then provide sample documents to train or fine-tune the extraction. The output plugs directly into an Appian Expression Rule or a Process Model.

**Whales:** So we need labelled sample documents to set this up. Do we have those from the ops team?

**Deepali:** We have about 40 anonymized invoices they shared last month. For contracts, we're still waiting — legal is reviewing what they can share. George, can you chase that dependency?

**Deepali:** Both are in scope long-term, but for MVP we're prioritizing native digital PDFs. Scanned documents introduce OCR variability and I don't want that noise in our first rollout. We'll tackle scanned docs in Phase 2 once the base flow is stable.

**Rahul:** Perfect, that simplifies the error states I need to design for. I'll still add a graceful error message if someone accidentally uploads a scanned file — something like "This document format isn't supported yet. Please upload a digital PDF."

**Rahul:** Agreed. My scope is the extraction review interface, the field mapping admin panel, and the error/empty states. I'll have low-fidelity wireframes ready by end of Sprint 1, Day 3 for review.

And I'll set up the Appian AI Skill environment in our dev instance so Rahul can start seeing real extraction output by mid-sprint. Any blockers anyone wants to call out?

**Whales:** My only concern is environment access. I need to confirm with IT that our dev Appian instance has the AI Skills feature enabled — it's a licensed add-on and I'm not 100% sure it's active yet.

**Deepali:** One last thing — let's make sure we demo the extraction flow to the ops team lead before Sprint 2 ends. Early buy-in from the actual users will save us a lot of rework later.
